**0.6.0 — Go CLI, workflow YAML (multi-step + async groups), resolvers, worktree on host, commit on host, template init, and Windows support (native `dockpipe.exe` + optional WSL bridge).**

**Run → Isolate → Act.** One primitive; workflows optional sugar on top.

---

## What's new

### CLI & codebase

- **Go is the default CLI** — `cmd/dockpipe` plus **`lib/dockpipe/`** in three layers: **`domain`** (workflow model, YAML parse, env rules), **`application`** (flags, `init` / `action` / `pre` / `template`, run orchestration), **`infrastructure`** (docker, bash pre-scripts, FS, git on host). The **`bin/dockpipe`** launcher runs **`bin/dockpipe.bin`** when built with `make`, otherwise `go run ./cmd/dockpipe`. Legacy all-bash path remains under `scripts/` / `lib/*.sh` for reference, not the default.
- **Native YAML for workflows** — `config.yml` is parsed with **`gopkg.in/yaml.v3`** in Go. Multi-step pipelines do **not** depend on python3.
- **Automated release pipeline** — GitHub Actions workflow builds release artifacts (Linux/macOS/Windows), `.deb`, checksums, and publishes GitHub releases from `releasenotes/<version>.md`. Blog publishing remains manual.

### Multi-step workflows (`steps:`)

- **Ordered steps** — Each step can set `cmd` / `command`, `isolate`, `run` / `pre_script`, `act`, `vars`, `outputs` (dotenv file handoff to later steps), and `skip_container`.
- **Async / parallel groups** — Consecutive steps with **`is_blocking: false`** run concurrently; the **next blocking** step is the **join** and receives env after all outputs are merged in **declaration order** (same key → **last** wins).
- **Optional `group` syntax** — `- group: { mode: async, tasks: [...] }` expands to the same runtime as flat `is_blocking: false` steps (readability only).
- **Merge transparency** — When parallel outputs overwrite a key, stderr logs: `[dockpipe] [merge] variable "…" overwritten by … (previously set by …)`.
- **Optional step `id`** — Labels steps in logs and merge messages.

Full user contract: **[docs/workflow-yaml.md](../docs/workflow-yaml.md)**.

### Resolvers, worktree, commit (from earlier 0.6 planning)

- **Resolvers** — One file per tool in `templates/.../resolvers/` (e.g. `claude`, `codex`). Use `--resolver <name>` with `--repo` / `--branch`; add files without core code changes.
- **Worktree on host** — `--repo <url>` and `--branch`: clone/worktree on the **host**, mount at `/work`, run the command in the container.
- **Commit on host** — Bundled **commit-worktree** (and `--repo` / `--branch` flows) run **git on the host** after the container exits.
- **Template init** — `dockpipe template init my-ai [--from init|run|test|…]` copies a bundled workflow folder. Default **`--from`** is **`init`**. Use **`strategy: worktree`** in your **`config.yml`** for resolver + clone flows — see **workflow-yaml.md**.

### Windows (Docker Desktop + native CLI)

- **Default: run on Windows** — `dockpipe.exe` uses **Docker Desktop**, **Windows git**, and **bash** on PATH (e.g. Git for Windows) for pre-scripts; no WSL required for normal use.
- **Optional WSL bridge** — set **`DOCKPIPE_USE_WSL_BRIDGE=1`** to forward the CLI into a distro: cwd mapped with **`wslpath`**, **path-like flags** (`C:\…`, UNC, etc.) rewritten for the inner Linux argv; the user command after **`--`** is not rewritten. **Unit tests** cover mount splitting, `--` preservation, subcommands, and **UNC-safe** fallback when `wslpath` is unavailable (e.g. Linux CI).
- **`windows` command group** — `dockpipe windows setup` and `dockpipe windows doctor` (for bridge users: pick distro, persist `%APPDATA%\dockpipe\windows-config.env`, optional `--install-command` to install Linux `dockpipe` in the distro).
- **WSL bootstrap for host-aware flows** — setup writes `~/.dockpipe/windows-host.env` in the chosen distro and wires it into `.bashrc` (includes `DOCKPIPE_WINDOWS_HOST=1` and `DOCKPIPE_WINDOWS_FETCH_BACK=1`).

## Installation

Full reference: **[docs/install.md](https://github.com/jamie-steele/dockpipe/blob/v0.6.0/docs/install.md)**. Below: **Linux**, **macOS**, **Windows** for this tag (**v0.6.0**).

### Linux

**Prerequisites**

- **Docker** (`docker.io` / Docker CE, etc.) — required.
- **Bash** on the host — required (dockpipe always invokes bash).
- **git** on the host — for **clone / worktree / commit-on-host** only (e.g. `--repo`, `clone-worktree.sh`).

**Option A — `.deb` (recommended)**

Pick the package that matches your CPU (**amd64** = x86_64, **arm64** = aarch64). They are not interchangeable.

**x86_64:**

```bash
wget https://github.com/jamie-steele/dockpipe/releases/download/v0.6.0/dockpipe_0.6.0_amd64.deb
sudo dpkg -i dockpipe_0.6.0_amd64.deb
```

**aarch64 (ARM64):**

```bash
wget https://github.com/jamie-steele/dockpipe/releases/download/v0.6.0/dockpipe_0.6.0_arm64.deb
sudo dpkg -i dockpipe_0.6.0_arm64.deb
```

If `dpkg` reports missing dependencies:

```bash
sudo apt-get install -f
```

**Option B — tarball** (single static binary, no `dpkg`)

```bash
# amd64
wget https://github.com/jamie-steele/dockpipe/releases/download/v0.6.0/dockpipe_0.6.0_linux_amd64.tar.gz
tar -xzf dockpipe_0.6.0_linux_amd64.tar.gz
sudo install -m 0755 dockpipe /usr/local/bin/dockpipe
```

```bash
# arm64
wget https://github.com/jamie-steele/dockpipe/releases/download/v0.6.0/dockpipe_0.6.0_linux_arm64.tar.gz
tar -xzf dockpipe_0.6.0_linux_arm64.tar.gz
sudo install -m 0755 dockpipe /usr/local/bin/dockpipe
```

**Option C — build from source**

Requires **Go** (see repo **`go.mod`** for the toolchain). From a clone at **`v0.6.0`**:

```bash
git clone https://github.com/jamie-steele/dockpipe.git
cd dockpipe && git checkout v0.6.0
make   # or: go build -o bin/dockpipe.bin ./cmd/dockpipe
export PATH="$PATH:$(pwd)/bin"
```

### macOS

**Prerequisites**

- **Docker Desktop for Mac** (or another Docker engine) — required.
- **Bash** — required (`/bin/bash` is typical).
- **git** — for worktree / `--repo` / commit-on-host flows.

**Option A — Homebrew** (after the tap is published)

```bash
brew tap jamie-steele/dockpipe
brew install dockpipe
```

Upgrade: `brew update && brew upgrade dockpipe`.

**Option B — release tarball** (Apple Silicon vs Intel)

**Apple Silicon (arm64):**

```bash
curl -LO https://github.com/jamie-steele/dockpipe/releases/download/v0.6.0/dockpipe_0.6.0_darwin_arm64.tar.gz
tar -xzf dockpipe_0.6.0_darwin_arm64.tar.gz
sudo install -m 0755 dockpipe /usr/local/bin/dockpipe
```

**Intel (amd64):**

```bash
curl -LO https://github.com/jamie-steele/dockpipe/releases/download/v0.6.0/dockpipe_0.6.0_darwin_amd64.tar.gz
tar -xzf dockpipe_0.6.0_darwin_amd64.tar.gz
sudo install -m 0755 dockpipe /usr/local/bin/dockpipe
```

**Option C — build from source**

```bash
git clone https://github.com/jamie-steele/dockpipe.git
cd dockpipe && git checkout v0.6.0
make
export PATH="$PATH:$(pwd)/bin"
```

To persist `PATH` on zsh (run from the clone):

```bash
echo "export PATH=\"\$PATH:$(pwd)/bin\"" >> ~/.zshrc
```

### Windows

**Prerequisites**

- **Docker Desktop** — required.
- **`bash.exe`** on `PATH` — required (dockpipe always uses bash on the host). **Git for Windows** is the usual install ( **`bash.exe` + `git.exe`** ). Docker Desktop does **not** ship bash.
- **`git`** — additionally required for **worktrees**, **`--repo`**, **commit-on-host**, and similar flows.

**Option A — PowerShell install script** (checksums when available, per-user `PATH`)

Latest from `master` (follows latest release on GitHub):

```powershell
irm https://raw.githubusercontent.com/jamie-steele/dockpipe/master/packaging/windows/install.ps1 | iex
```

**This version (0.6.0) explicitly** — save the script and pass **`-Version`**:

```powershell
$i = "$env:TEMP\dockpipe-install.ps1"
Invoke-WebRequest -Uri "https://raw.githubusercontent.com/jamie-steele/dockpipe/master/packaging/windows/install.ps1" -OutFile $i -UseBasicParsing
& $i -Version 0.6.0
```

Open a **new** terminal after install so `PATH` updates.

**Option B — MSI (coming soon)**

Per-user WiX installer (`dockpipe_<version>_windows_amd64.msi`, adds `%LOCALAPPDATA%\dockpipe` to user **PATH**) is **not** published for **v0.6.0** — it was cut from this release. Use **Option A** (script) or **Option C** (zip). MSI will ship in a future release when the WiX pipeline is enabled for that tag (see **`packaging/msi/README.md`**).

**Option C — zip**

Download **`dockpipe_0.6.0_windows_amd64.zip`**, unzip, add the folder to **PATH** (or invoke `dockpipe.exe` by full path). New terminal after `PATH` changes.

**Verify**

```powershell
dockpipe --version
dockpipe -- echo ok
```

**Optional — WSL bridge** (Linux `dockpipe` inside a distro; not required for native Windows use)

Install **`dockpipe`** inside WSL (e.g. `.deb` or build), then on Windows:

```powershell
dockpipe windows setup
dockpipe windows doctor
```

Enable forwarding for a session:

```powershell
$env:DOCKPIPE_USE_WSL_BRIDGE = "1"
```

See **[docs/wsl-windows.md](https://github.com/jamie-steele/dockpipe/blob/v0.6.0/docs/wsl-windows.md)** for bundle / mixed-clone notes.

**Build from source on Windows** (Go per **`go.mod`**):

```powershell
git clone https://github.com/jamie-steele/dockpipe.git
cd dockpipe
git checkout v0.6.0
$env:GOOS = "windows"; $env:GOARCH = "amd64"; $env:CGO_ENABLED = "0"
go build -trimpath -ldflags "-s -w -X main.Version=0.6.0" -o dockpipe.exe ./cmd/dockpipe
```

Add **`dockpipe.exe`** to `PATH` or run from the build directory.

## Upgrade from 0.5.x

1. Install the new `.deb` (or rebuild from source).
2. **Workflows:** If you relied on bash-only multi-step helpers, prefer **`steps:`** in `config.yml` as documented in **`docs/workflow-yaml.md`**.
3. **Images:** Rebuild template images if you use **versioned** image tags from `version`.
4. New flags / behavior: **`--resolver`**, **`--repo`**, **`--branch`**, **`dockpipe template init`**, multi-step **`is_blocking`** / **`group.mode: async`**.

## Examples

**AI in worktree, commit on host:**

```bash
dockpipe --resolver claude --repo https://github.com/you/repo.git --branch claude/task \
  -- claude --dangerously-skip-permissions -p "Fix the bug"
```

**Copy a workflow template:**

```bash
dockpipe template init my-ai --from init
dockpipe --workflow my-ai --repo https://github.com/you/repo.git -- claude -p "Fix the bug"
```

**Multi-step + async (see `docs/workflow-yaml.md` for full syntax):**

```yaml
# templates/my-flow/config.yml
steps:
  - id: lint
    cmd: make lint
    is_blocking: true
  - group:
      mode: async
      tasks:
        - id: test
          cmd: make test
          outputs: .dockpipe/test.env
        - id: audit
          cmd: make audit
          outputs: .dockpipe/audit.env
  - id: report
    cmd: sh -c 'echo done'
    is_blocking: true
```

```bash
dockpipe --workflow my-flow
```

(`steps:` mode does not require `--` when every step defines `cmd` / `command`; you can still pass extra argv for the last step if its command is empty — see **docs/workflow-yaml.md**.)

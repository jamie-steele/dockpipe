**0.6.0 ‚Äî Go CLI, workflow YAML (multi-step + async groups), resolvers, worktree on host, commit on host, template init, and Windows support (native `dockpipe.exe` + optional WSL bridge).**

**Run ‚Üí Isolate ‚Üí Act.** One primitive; workflows optional sugar on top.

---

## What's new

### CLI & codebase

- **Go is the default CLI** ‚Äî `cmd/dockpipe` plus **`lib/dockpipe/`** in three layers: **`domain`** (workflow model, YAML parse, env rules), **`application`** (flags, `init` / `action` / `pre` / `template`, run orchestration), **`infrastructure`** (docker, bash pre-scripts, FS, git on host). The **`bin/dockpipe`** launcher runs **`bin/dockpipe.bin`** when built with `make`, otherwise `go run ./cmd/dockpipe`. Legacy all-bash path remains under `scripts/` / `lib/*.sh` for reference, not the default.
- **Native YAML for workflows** ‚Äî `config.yml` is parsed with **`gopkg.in/yaml.v3`** in Go. Multi-step pipelines do **not** depend on python3.
- **Automated release pipeline** ‚Äî GitHub Actions workflow builds release artifacts (Linux/macOS/Windows), `.deb`, checksums, and publishes GitHub releases from `releasenotes/<version>.md`. Blog publishing remains manual.

### Multi-step workflows (`steps:`)

- **Ordered steps** ‚Äî Each step can set `cmd` / `command`, `isolate`, `run` / `pre_script`, `act`, `vars`, `outputs` (dotenv file handoff to later steps), and `skip_container`.
- **Async / parallel groups** ‚Äî Consecutive steps with **`is_blocking: false`** run concurrently; the **next blocking** step is the **join** and receives env after all outputs are merged in **declaration order** (same key ‚Üí **last** wins).
- **Optional `group` syntax** ‚Äî `- group: { mode: async, tasks: [...] }` expands to the same runtime as flat `is_blocking: false` steps (readability only).
- **Merge transparency** ‚Äî When parallel outputs overwrite a key, stderr logs: `[dockpipe] [merge] variable "‚Ä¶" overwritten by ‚Ä¶ (previously set by ‚Ä¶)`.
- **Optional step `id`** ‚Äî Labels steps in logs and merge messages.

Full user contract: **[docs/workflow-yaml.md](../docs/workflow-yaml.md)**.

### Resolvers, worktree, commit (from earlier 0.6 planning)

- **Resolvers** ‚Äî One file per tool in `templates/.../resolvers/` (e.g. `claude`, `codex`). Use `--resolver <name>` with `--repo` / `--branch`; add files without core code changes.
- **Worktree on host** ‚Äî `--repo <url>` and `--branch`: clone/worktree on the **host**, mount at `/work`, run the command in the container.
- **Commit on host** ‚Äî Bundled **commit-worktree** (and `--repo` / `--branch` flows) run **git on the host** after the container exits.
- **Template init** ‚Äî `dockpipe template init my-ai [--from llm-worktree]` copies workflow layout (config, resolvers, isolate). Config points at repo **scripts/**.

### Windows (Docker Desktop + native CLI)

- **Default: run on Windows** ‚Äî `dockpipe.exe` uses **Docker Desktop**, **Windows git**, and **bash** on PATH (e.g. Git for Windows) for pre-scripts; no WSL required for normal use.
- **Optional WSL bridge** ‚Äî set **`DOCKPIPE_USE_WSL_BRIDGE=1`** to forward the CLI into a distro: cwd mapped with **`wslpath`**, **path-like flags** (`C:\‚Ä¶`, UNC, etc.) rewritten for the inner Linux argv; the user command after **`--`** is not rewritten. **Unit tests** cover mount splitting, `--` preservation, subcommands, and **UNC-safe** fallback when `wslpath` is unavailable (e.g. Linux CI).
- **`windows` command group** ‚Äî `dockpipe windows setup` and `dockpipe windows doctor` (for bridge users: pick distro, persist `%APPDATA%\dockpipe\windows-config.env`, optional `--install-command` to install Linux `dockpipe` in the distro).
- **WSL bootstrap for host-aware flows** ‚Äî setup writes `~/.dockpipe/windows-host.env` in the chosen distro and wires it into `.bashrc` (includes `DOCKPIPE_WINDOWS_HOST=1` and `DOCKPIPE_WINDOWS_FETCH_BACK=1`).

## Installation

### Linux

**x86_64** ‚Äî `.deb`:

```bash
wget https://github.com/jamie-steele/dockpipe/releases/download/v0.6.0/dockpipe_0.6.0_amd64.deb
sudo dpkg -i dockpipe_0.6.0_amd64.deb
```

**aarch64 (ARM64)** ‚Äî use the **arm64** package (not the amd64 one):

```bash
wget https://github.com/jamie-steele/dockpipe/releases/download/v0.6.0/dockpipe_0.6.0_arm64.deb
sudo dpkg -i dockpipe_0.6.0_arm64.deb
```

Or use the matching **`linux_amd64`** / **`linux_arm64`** tarball from the same release if you prefer not to use `dpkg`.

Requires **Bash**, **Docker**, and **git** (for commit-on-host). If `dpkg` reports missing deps: `sudo apt install -f`.

**From source:** **Go 1.22+** ‚Äî `make` or `go build -o bin/dockpipe.bin ./cmd/dockpipe`.

### Windows

**Install the Windows CLI** (pick one):

- **Script (MSI + checksum):** `irm https://raw.githubusercontent.com/jamie-steele/dockpipe/master/packaging/windows/install.ps1 | iex`
- **MSI:** `dockpipe_0.6.0_windows_amd64.msi` from this release (per-user, updates PATH)
- **Zip:** `dockpipe_0.6.0_windows_amd64.zip` ‚Äî unzip and add to `PATH`

**Requirements:** **Docker Desktop** (containers only ‚Äî it does not install host `git` / `bash`) plus **Git for Windows** (or anything that puts `git` and `bash` on PATH). From PowerShell: `cd` to your repo and run `dockpipe` like on Linux.

**Optional ‚Äî run the Linux CLI inside WSL** (`DOCKPIPE_USE_WSL_BRIDGE=1`): install **`dockpipe`** in the distro (e.g. `.deb`), then:

```powershell
dockpipe windows setup
```

Non-interactive (explicit distro + install command):

```powershell
dockpipe windows setup --distro Ubuntu --install-command "<your install command>" --non-interactive
dockpipe windows doctor
```

### macOS

Supported from source, plus Homebrew formula packaging is now included in-repo:

```bash
git clone https://github.com/jamie-steele/dockpipe.git
cd dockpipe
make
export PATH="$PATH:$(pwd)/bin"
```

Homebrew path (after tap publication):

```bash
brew tap jamie-steele/dockpipe
brew install dockpipe
```

## Upgrade from 0.5.x

1. Install the new `.deb` (or rebuild from source).
2. **Workflows:** If you relied on bash-only multi-step helpers, prefer **`steps:`** in `config.yml` as documented in **`docs/workflow-yaml.md`**.
3. **Images:** Rebuild template images if you use **versioned** image tags from `version`.
4. New flags / behavior: **`--resolver`**, **`--repo`**, **`--branch`**, **`dockpipe template init`**, multi-step **`is_blocking`** / **`group.mode: async`**.

## Examples

**AI in worktree, commit on host:**

```bash
dockpipe --resolver claude --repo https://github.com/you/repo.git --branch claude/task \
  -- claude --dangerously-skip-permissions -p "Fix the bug"
```

**Copy a workflow template:**

```bash
dockpipe template init my-ai --from llm-worktree
dockpipe --workflow my-ai --repo https://github.com/you/repo.git -- claude -p "Fix the bug"
```

**Multi-step + async (see `docs/workflow-yaml.md` for full syntax):**

```yaml
# templates/my-flow/config.yml
steps:
  - id: lint
    cmd: make lint
    is_blocking: true
  - group:
      mode: async
      tasks:
        - id: test
          cmd: make test
          outputs: .dockpipe/test.env
        - id: audit
          cmd: make audit
          outputs: .dockpipe/audit.env
  - id: report
    cmd: sh -c 'echo done'
    is_blocking: true
```

```bash
dockpipe --workflow my-flow
```

(`steps:` mode does not require `--` when every step defines `cmd` / `command`; you can still pass extra argv for the last step if its command is empty ‚Äî see **docs/workflow-yaml.md**.)
